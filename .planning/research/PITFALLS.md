# Domain Pitfalls: .NET Microservices Codebase Hardening

**Domain:** Security hardening and test retrofitting for .NET microservices + React Native app
**Researched:** 2026-03-29
**Overall confidence:** HIGH (findings grounded in official ASP.NET Core docs, well-documented community patterns, and direct codebase analysis)

---

## Critical Pitfalls

Mistakes that cause rewrites, security regressions, or invalidate the hardening effort entirely.

---

### Pitfall 1: Fixing Secrets Without Rotating Them

**What goes wrong:** Removing a hardcoded credential from `appsettings.json` and replacing it with an environment variable placeholder. The secret is no longer in code — but it was already committed to git history, where it stays forever. The "fix" satisfies a code review but does not actually reduce the exposure.

**Why it happens:** Developers treat secret removal as a code change rather than an incident response. The mental model is "clean up the code"; the correct mental model is "this credential is compromised and must be treated as such."

**Consequences:** Hardcoded community-service DB password (`Password=postgres`) and the RunPod proxy URL are both in git history. Any clone of this repo — now or in the future — contains them. Even a private repo exposes them to all contributors, CI systems, and anyone who ever copies the config for debugging.

**Prevention:**
- Treat every committed secret as already-leaked. Rotate the credential first, then remove the hardcoded value.
- For the RunPod proxy URL specifically: if the RunPod instance is still running, that endpoint is exposed. Change the proxy configuration or rotate the token even if the URL looks benign.
- Set `appsettings.json` values to empty strings or obvious placeholders (e.g., `"CHANGE_ME"`) that will cause startup failures if not overridden — not silent empty-string bugs.
- Consider running `git filter-repo` or BFG Repo Cleaner to scrub the historical commits if this will be a public or evaluated repo.

**Detection:** Run `git log --all -S "Password=postgres"` and `git log --all -S "runpod.net"`. Any output means the secret lives in history.

**Phase mapping:** Security phase (first phase). Do not defer to a cleanup pass.

### Pitfall 2: Removing the 2FA Log Line but Leaving the Vulnerability Window

**What goes wrong:** Line 125 of `auth-service/Services/AuthService.cs` logs the 2FA code. The obvious fix is to delete that line. But the underlying generator (`GenerateRandomCode()` using `new Random()`) is also broken. Teams fix one symptom (logging) without fixing the other (predictability), leaving the 2FA system compromised from a different angle.

**Why it happens:** CONCERNS.md lists them as two separate line-item fixes. It is tempting to close them independently. Predictable codes do not produce an obvious runtime error — the system appears to work correctly.

**Consequences:** Even with the log line removed, an attacker with enough API requests can predict the 6-digit code generated by `System.Random` because the RNG seed is based on system time and is not cryptographically random. This effectively means 2FA provides no additional security against a targeted attack.

**Prevention:** Fix both items as a single atomic change. The correct fix is:
1. Remove the log statement.
2. Replace `new Random().Next(100000, 1000000)` with `RandomNumberGenerator.GetInt32(100000, 1000000)`.
3. Write a test that verifies the generated codes are not sequentially predictable (e.g., generate 1000 codes, assert no run of 10 shares a predictable pattern — or simply verify the method uses `RandomNumberGenerator` by type reflection or a wrapper interface).

**Detection:** Search for `new Random()` across the entire auth-service. Any hit in a security context is a flag.

**Phase mapping:** Security phase. Both fixes must be in the same commit/PR.

### Pitfall 3: Community Service Authentication Added Incorrectly — Gateway Check Placed After Request Handling

**What goes wrong:** Adding the gateway shared-secret middleware to community-service but registering it in the wrong position in the middleware pipeline. If it is placed after `app.MapControllers()` or after routing resolves, the check never executes before the controller handles the request.

**Why it happens:** ASP.NET Core middleware order is unintuitive. The community-service currently has no middleware for authentication, so there is no existing pattern in that service to follow. Developers new to YARP gateway patterns may not know the correct position.

**Consequences:** The endpoint appears protected (middleware is registered), but controllers execute before the check runs. Any user who bypasses the YARP gateway continues to impersonate any userId by forging the `X-User-Id` header.

**Prevention:**
- Follow the established pattern from `notification-service` and `journal-service` exactly. Look at how they register their gateway middleware relative to routing and authorization.
- Correct ASP.NET Core order: `UseRouting()` → `UseCors()` → `UseAuthentication()` → `UseAuthorization()` → custom middleware → `MapControllers()`. Gateway shared-secret checks should be before `MapControllers()`.
- Write an integration test that sends a request without the shared secret and asserts a 401 response. This test will catch a misregistration immediately.

**Detection:** After implementing, send a request to the community-service directly (bypassing the auth gateway) without the `X-Gateway-Secret` header. If the request succeeds, the middleware is in the wrong position.

**Phase mapping:** Security phase.

### Pitfall 4: Exception Message Exposure Fix Breaks Error Handling for Known Business Errors

**What goes wrong:** The fix for exception messages exposed to clients (`return BadRequest(new { message = ex.Message })`) is to return a generic message. But `AuthController` currently relies on `ex.Message` to distinguish between different `Exception` thrown by `AuthService` (e.g., "Email already registered", "Account is locked"). If you swap all `ex.Message` returns for `"An error occurred"`, the frontend loses the specific error context it needs to display meaningful messages to users.

**Why it happens:** The PII exposure fix and the domain exceptions fix (generic `Exception` types) are listed as separate items in CONCERNS.md at different priorities. The security fix gets done first; the domain exceptions fix is deferred. This leaves the codebase in a broken state where secure equals non-functional.

**Consequences:** Registration returns a generic error even for "email already exists". Users get no useful feedback. The app appears broken rather than hardened.

**Prevention:** Treat these two items as a coupled fix:
1. Create domain exception types: `DuplicateEmailException`, `AccountLockedException`, `InvalidCredentialsException`, etc.
2. Update `AuthService` to throw these typed exceptions.
3. Update `AuthController` to catch typed exceptions and return appropriate HTTP status codes with safe messages (e.g., `DuplicateEmailException` → 409 Conflict with a safe generic message, not `ex.Message`).
4. Only at that point return generic messages for the catch-all `Exception`.

**Detection:** After fixing exception exposure, manually test the registration flow with a duplicate email. If the UI shows "An error occurred" instead of "Email already in use", the coupling has been broken.

**Phase mapping:** Security phase (exception exposure) must coordinate with Tech Debt phase (domain exceptions). Plan them as a single unit of work.

## Moderate Pitfalls

Mistakes that cause test coverage to be misleading or fixes to be incomplete, without immediate security consequences.

---

### Pitfall 5: Writing Tests That Encode the Bug as Expected Behavior (Characterization Tests)

**What goes wrong:** When retrofitting tests onto existing untested code, the fastest approach is to run the code, observe its output, and write assertions matching the current (buggy) behavior. This produces tests that pass immediately but test the wrong thing. For example, writing a test for `GetUserInfoByEmail` that passes any valid JWT — regardless of whose email is in the URL — because that is the current behavior.

**Why it happens:** The pressure to get test coverage numbers up quickly. It is genuinely hard to know, when looking at unfamiliar code, whether the current behavior is correct or a bug. The codebase has zero tests for auth-service; any tests feel like progress.

**Consequences:** Tests pass the moment they are written and continue passing after hardening — because the test is asserting the vulnerability, not preventing it. Coverage metrics look good; the vulnerability remains. This is sometimes called "test debt" — tests that add noise and false confidence rather than real protection.

**Prevention:**
- For each fix in CONCERNS.md, write the test first (RED), then implement the fix so the test passes (GREEN). This is the only way to be certain the test is testing the fix.
- Specifically for `GetUserInfoByEmail`: write a test that authenticates as User A, requests `/api/auth/user-info/{user_b_email}`, and asserts 403. This test must fail before the fix and pass after.
- Never write a test that passes against the current broken code, mark it as passing, and call the fix "proven."

**Detection:** If a newly written test passes without any code change, it is a characterization test. Stop and reconsider what it is actually asserting.

**Phase mapping:** Every phase with security fixes. Especially critical in the auth-service test project creation.

### Pitfall 6: Auth Service Tests Require Real BCrypt Work Factor — Test Suite Becomes Slow

**What goes wrong:** Auth service tests that exercise `AuthService.RegisterAsync()` or `AuthService.AuthenticateAsync()` will run BCrypt password hashing at the default work factor (configured in the service). BCrypt at work factor 12 takes ~300ms per hash. A test suite with 20 auth flow tests runs in 6+ seconds just for hashing — before any database I/O.

**Why it happens:** `AuthService` directly calls `BCrypt.Net.BCrypt.HashPassword()` without an abstraction layer. Integration tests that spin up the real service also spin up real BCrypt.

**Consequences:** Tests are slow, developers skip running them locally, CI times balloon, and the test suite becomes something teams avoid rather than rely on.

**Prevention:**
- Wrap BCrypt behind an `IPasswordHasher` interface (this is the ASP.NET Core Identity pattern).
- In unit tests, inject a fast mock/fake that returns a predictable string.
- In integration tests that need real password verification, use work factor 4 (the minimum) via test configuration override.
- Apply this pattern when creating the auth-service test project — do not wait for a performance complaint.

**Detection:** Time the test suite. If auth tests take >500ms per test, BCrypt work factor is not being controlled.

**Phase mapping:** Test coverage phase for auth-service.

### Pitfall 7: Chat-Service Integration Tests Require Live Database — Tests Fail in CI

**What goes wrong:** The existing `ChatDatabaseProvider.IntegrationTests.cs` uses a hardcoded `StubConfigurationService` with a connection string pointing to `localhost:5432`. New tests added following the same pattern will require a real PostgreSQL instance, making the test suite impossible to run in isolation and incompatible with any future CI pipeline.

**Why it happens:** The existing test is marked as an anti-pattern in TESTING.md but is the only example in the chat-service test project. When developers add new tests, they follow the only example available.

**Consequences:** Test suite cannot run without Docker Compose running. New developers cannot run tests. Tests pass locally but fail in any environment without the database. Coverage metrics are meaningless because tests are routinely skipped.

**Prevention:**
- Do not extend the hardcoded connection string pattern. It is documented as an anti-pattern in TESTING.md.
- New chat-service tests must follow the `WebApplicationFactory` + in-memory or testcontainers pattern established in `AI-Wrapper-Service`.
- For `SessionService` unit tests (immutability fix, bookmark logic), use Moq to mock `ISessionDatabaseProvider` and `ICacheServiceProvider`. No database needed.
- For `ChatController` integration tests, use `WebApplicationFactory<Program>` with mocked database providers.

**Detection:** If a test file contains a hardcoded connection string, it will break outside the developer's machine.

**Phase mapping:** Test coverage phase for chat-service.

### Pitfall 8: Rate Limiting Middleware Fix Introduces a Race Condition During Migration

**What goes wrong:** The auth-service rate limiting middleware must be moved to after `UseCors()` and `UseAuthentication()`. During this reordering, it is possible to accidentally remove the rate limiting registration entirely (by misreading which `app.Use*` call to move), or to move it to a position where CORS preflight requests are still counted.

**Why it happens:** The middleware registration in `auth-service/Program.cs` is a sequence of `app.Use*` calls. Moving one line is syntactically trivial but semantically significant. Without a test verifying the behavior, the reordering is unverified.

**Consequences:** Legitimate CORS preflight requests continue to consume rate limit budget (false positives for the browser). Or, worse, rate limiting is silently disabled.

**Prevention:**
- Write an integration test for rate limiting behavior before touching the middleware order. The test should verify that a request that exceeds the limit gets a 429 response.
- After reordering, re-run the test. If it still passes, the reorder did not break the rate limiter.
- Verify CORS preflight (OPTIONS request) is not counted against the rate limit after the fix.

**Detection:** Send an OPTIONS preflight request after the fix. It should not trigger rate limiting.

**Phase mapping:** Bugs/error-prone areas phase.

### Pitfall 9: StoredProcedureExecutor Whitelist Fix Breaks Callers Using Qualified Names

**What goes wrong:** Adding a regex whitelist validation of `[a-zA-Z0-9_.]` to `BuildFunctionCallSql()` in the stored procedure executor seems straightforward. But PostgreSQL function calls can use schema-qualified names (e.g., `public.get_journal_entries`) which include a period. If the whitelist only allows `[a-zA-Z0-9_]` (no dot), callers using schema-qualified names break silently with a generic validation error.

**Why it happens:** The fix is written without checking how every caller currently passes function names.

**Consequences:** Journal or notification service calls start returning 500 errors. Since there are no tests for these services, the regression is silent until someone notices missing data.

**Prevention:**
- Before writing the whitelist, audit all callers of `StoredProcedureExecutor.ExecuteAsync()` across both services. Note whether any pass schema-qualified names.
- Include the period in the whitelist pattern: `^[a-zA-Z0-9_.]+$`.
- Write a parameterized test covering both valid and invalid names (e.g., `"public.get_journal"` should pass; `"get_journal; DROP TABLE"` should fail).

**Detection:** After the fix, run all journal-service and notification-service endpoints. Any 500 error on a previously working endpoint indicates a whitelist regression.

**Phase mapping:** Bugs/error-prone areas phase. Coordinate with the duplication-elimination tech debt item.

## Minor Pitfalls

Issues that waste time or reduce quality without critical consequences.

---

### Pitfall 10: WebApplicationFactory Shared State Across Test Classes Causes Flaky Tests

**What goes wrong:** When multiple test classes share a `WebApplicationFactory` instance (via `IClassFixture`), state from one test leaks into another — particularly for stateful middleware like rate limiting. The `IsolatedWebApplicationFactory` pattern used in `AI-Wrapper-Service` tests exists precisely to avoid this, but is easy to overlook when creating new test projects from scratch.

**Prevention:**
- Use `IClassFixture<CustomWebApplicationFactory<Program>>` for tests that share state-free factory.
- Use `IsolatedWebApplicationFactory` (new instance per test) for anything involving rate limiting, caching, or middleware state.
- Do not use `ICollectionFixture` across services — each service's test suite should be independent.

**Phase mapping:** Test coverage phase, especially when setting up auth-service and community-service test projects.

### Pitfall 11: React Native / Jest Setup Swallows Async AuthContext Failures Silently

**What goes wrong:** The frontend has no test framework configured. When Jest + React Native Testing Library is added, tests for `AuthContext.js` that involve async operations (token load from `expo-secure-store`, API calls) commonly fail with `act()` warnings or pass despite incorrect behavior because assertions run before async state settles.

**Prevention:**
- Wrap all test renders in the real `AuthProvider` (or a test-scoped wrapper that provides the same context shape).
- Use `waitFor` or `findBy*` queries for any assertion on state that changes after an async operation — never use `setTimeout`.
- Mock `expo-secure-store` at the module level (not inline) since it is a native module that has no Jest implementation.
- Mock `api.js` and `chatApi.js` at the service boundary, not the `fetch` level, to avoid brittle implementation coupling.

**Detection:** Any test that passes immediately after async operations are triggered (without `waitFor`) is likely not testing the final state.

**Phase mapping:** Frontend test coverage phase.

### Pitfall 12: Docker Compose Port Exposure Fix Removes Ports Needed for Local Development

**What goes wrong:** Removing the host port mappings from `docker-compose.yml` for databases and Redis breaks local development workflows where developers connect directly to the database with tools like DBeaver, pgAdmin, or Redis Insight. The fix is correct for production but must not be applied to the development compose file without an alternative.

**Prevention:**
- Do not remove ports from the default `docker-compose.yml`. Instead, create a `docker-compose.prod.yml` override that removes the port mappings, or annotate the existing file with a comment that ports should be removed when deploying to production.
- Alternatively, use a `.env`-controlled approach where port exposure is conditional.

**Detection:** After the fix, verify that `dotnet test` still works for chat-service integration tests that require a real database. If they break, port removal was too aggressive.

**Phase mapping:** Configuration/deployment phase (last phase).

### Pitfall 13: Wearable Service Removal Creates Orphaned Imports and Silent UI Breakage

**What goes wrong:** The wearable service stub (`frontend/src/services/wearableService.js`) is imported by `WearableSettingsScreen.js`. Simply deleting the service file without removing the screen, the navigation entry that points to it, and the settings menu item that links to it will cause a runtime JavaScript import error — the app crashes on launch or on navigation.

**Prevention:**
- Before deleting the file, search for all imports: `import.*wearableService` and `WearableSettingsScreen` references in navigation and settings files.
- Remove in dependency order: navigation entry → screen component → service file.
- Verify the app launches and settings navigation works after removal.

**Detection:** A runtime `Cannot find module './wearableService'` error on any screen that imports it.

**Phase mapping:** Tech debt phase.

## Phase-Specific Warnings

| Phase Topic | Likely Pitfall | Mitigation |
|---|---|---|
| 2FA plaintext log fix | Fixing log without fixing `new Random()` (Pitfall 2) | Treat as a single atomic change |
| Exception exposure fix | Breaking frontend error messages (Pitfall 4) | Co-fix with domain exceptions |
| Community auth middleware | Wrong middleware position (Pitfall 3) | Write test before implementing; verify directly without gateway |
| Removing hardcoded secrets | Not rotating already-committed credentials (Pitfall 1) | Rotate first, remove second |
| Auth service test project | BCrypt slowness in tests (Pitfall 6) | Abstract behind `IPasswordHasher`; use work factor 4 in tests |
| Auth service test project | Characterization tests encoding bugs (Pitfall 5) | Write tests against fix target, not current behavior |
| Chat service test project | Extending live-DB anti-pattern (Pitfall 7) | Use WebApplicationFactory + mocked providers |
| Rate limiting middleware reorder | Accidentally disabling rate limiting (Pitfall 8) | Test rate limiting behavior before and after reorder |
| StoredProcedureExecutor whitelist | Breaking schema-qualified function names (Pitfall 9) | Audit callers; include dot in whitelist regex |
| Frontend test setup | Async AuthContext assertions (Pitfall 11) | Use `waitFor`; mock native modules at module level |
| Docker port cleanup | Breaking local dev database access (Pitfall 12) | Use override file, not main compose file |
| Wearable service removal | Orphaned imports crashing the app (Pitfall 13) | Search all import sites before deleting |

    **CFG-03 — Firebase fail-fast:**
    In `notification-service/src/NotificationService.Api/Program.cs`, after the existing try/catch block that calls `firebaseService.Initialize()` (lines 74-83), add a fail-fast check:

    ```csharp
    // Fail fast if Firebase path was configured but initialization failed
    var firebasePath = builder.Configuration["Firebase:ServiceAccountPath"];
    if (!string.IsNullOrWhiteSpace(firebasePath) && !firebaseService.IsInitialized)
    {
        throw new InvalidOperationException(
            $"Firebase:ServiceAccountPath is configured ('{firebasePath}') but initialization failed. " +
            "Ensure the file exists and is a valid service account JSON. " +
            "To run without Firebase, leave Firebase:ServiceAccountPath unset or empty.");
    }
    ```

    This preserves the optional-Firebase design: if the path is NOT set, the service starts normally using Expo Push. If the path IS set but the file is missing/invalid, the service aborts immediately with a clear error.

    **CRITICAL — Fix test factories to prevent startup failures:**
    The main `appsettings.json` sets `Firebase:ServiceAccountPath` to `"firebase-service-account.json"`. Since `AddInMemoryCollection` overlays on top of `appsettings.json`, the test factories currently inherit this non-empty value. After the fail-fast check is added, all notification-service tests will crash at startup because the file does not exist in the test environment.

    You MUST add `["Firebase:ServiceAccountPath"] = ""` to the in-memory config dictionary of BOTH test factories in `notification-service/src/NotificationService.Tests/Controllers/NotificationCodeControllerTests.cs`:

    1. In `NotificationTestFactory.ConfigureWebHost()`, add to the `AddInMemoryCollection` dictionary:
       ```csharp

       ```

    This is unconditionally required — both factories MUST override this key to empty string.
